# DIAGNOSIS — Color Emoji in the Caption Burn Pipeline

**Status:** Diagnosis only. No app code changed. Test artifacts live in the
session scratch dir (`.../scratchpad/emoji_test/`); this file is the report.

**Question:** When an emoji (🔥 😂 💯 🎉) appears in caption text, what does our
burn pipeline actually output — color emoji, monochrome outline, blank box, or
tofu?

**Short answer:** **Monochrome, single-color glyph outlines — tinted by the
karaoke color of the moment.** Not color emoji. The emoji shape is recognizable
(flame, laughing face, "100", party popper) but flat: it renders in whatever
color the word currently has (yellow when it is the highlighted word, white when
unspoken, grey when already-spoken), and it visibly *changes color* as the
karaoke sweep passes over it. And this "legible monochrome" result is
**specific to the current Windows host** — on a stock Linux host the same
pipeline produces **blank or tofu** (see the environment matrix below).

---

## 1. How the test was run (the real production path, not a generic test)

- ASS generated by the real `services.caption_renderer.generate_ass_karaoke(...)`
  with production defaults (`bold-yellow`, `Noto Sans Telugu`, 1080×1920). Caption
  text: `మీరు 🔥 mind-ni 😂` then `oka 💯 super 🎉` (emoji mixed with Telugu +
  Latin, exactly as specified).
- Burned by the real `services.caption_renderer.burn_captions(...)` — the exact
  production FFmpeg invocation: `-vf ass='<escaped>':fontsdir='<escaped>'`,
  single-pass, `libx264 -crf 23 -preset fast`, `fontsdir` =
  `services/assets/fonts/captions` (the bundled Telugu fonts).
- The generated ASS wraps each emoji in the same `{\1c..&\1a..&}` karaoke color
  override as every other word — nothing special happens to emoji at the ASS
  layer. `_escape_ass_text` leaves emoji untouched (no `{ } \` newline), so they
  pass through verbatim.

Frames extracted at each emoji's highlight moment are in the scratch dir
(`frame_prod_fire.png`, `frame_prod_hundred.png`, etc.).

**What rendered (per emoji):**

| Emoji | Rendered as | Color |
|-------|-------------|-------|
| 🔥 | flame **outline** (Segoe UI Emoji base layer) | flat — yellow while highlighted, white/grey otherwise |
| 😂 | laughing-face **outline** | flat — follows karaoke color |
| 💯 | "100" + underline **outline** | flat — follows karaoke color |
| 🎉 | party-popper **outline** | flat — follows karaoke color |

All four are legible silhouettes, all four are **single flat color**, none are
color emoji. Telugu (`మీరు`) and Latin (`mind-ni`, `super`) render correctly and
are unaffected.

## 2. Why — libass / FFmpeg findings (from `-v verbose`)

```
libass API version: 0x1704000
libass source: commit 0.17.4-16-ge60dddb7
Raster: FreeType 2.14.1
Shaper: FriBidi 1.0.16 (SIMPLE)  HarfBuzz-ng 12.2.0 (COMPLEX)
Using font provider directwrite (with GDI)
fontselect: (Noto Sans Telugu, 700, 0) -> NotoSansTelugu-Regular
Glyph 0x1F525 not found, selecting one more font for (Noto Sans Telugu, 700, 0)
fontselect: (Noto Sans Telugu, 700, 0) -> SegoeUIEmoji
```

- **FFmpeg 8.0.1** (gyan.dev full build), **libass 0.17.4**, FreeType 2.14.1.
- The bundled Telugu font has no emoji glyph (`Glyph 0x1F525 not found`), so
  libass falls back **past** `fontsdir` to a system font via the
  **directwrite** provider and picks **Segoe UI Emoji**
  (`C:\Windows\Fonts\seguiemj.ttf`, a **COLR/CPAL v0** color font).
- **libass 0.17.4 has no color-glyph compositing.** It rasterizes glyph
  *outlines* only (FreeType outline path, no `FT_LOAD_COLOR`), then fills them
  with the ASS text color. So it takes Segoe's base/outline layer and paints it
  the caption color. This is the direct, empirical proof for the version-support
  question in §4a — the color font is *loaded and used*, and still comes out
  monochrome.
- Note the provider is **directwrite**, not fontconfig, even though FFmpeg was
  built `--enable-fontconfig`. On this Windows host libass prefers DirectWrite,
  which is why a system emoji font is found at all. That is the crux of the
  environment dependency below.

## 3. Environment matters — this is NOT the same everywhere

The caption burn runs in the **worker on the host** (per `docker-compose.yml`,
only `redis` and `indicxlit` are containerized; the API/worker run on the host).
So the current deployment host *is* this Windows machine, and the "legible
monochrome" result above is what it produces today. But the fallback is
host-font-dependent, so the output changes with the host:

| Host / fallback font available | libass provider | Result for 🔥 |
|---|---|---|
| **Windows + Segoe UI Emoji** (current host) | directwrite | **monochrome outline**, karaoke-tinted, legible |
| Linux + **Noto Color Emoji** (CBDT bitmaps, common) | fontconfig | **blank / invisible** — CBDT glyphs have no outline for libass to trace |
| Linux + a monochrome fallback (e.g. DejaVu) with no emoji coverage | fontconfig | **tofu** (`.notdef` box) or dropped |
| Any host, no emoji-capable font at all | — | **tofu / dropped** |

So "what renders" is not one answer — it's monochrome-legible on the current
Windows host, and blank/tofu on a typical Linux container. Any fix must remove
this host dependency (bundle the emoji font in `fontsdir`), not just accept the
current host's accidental behavior.

## 4. Options (investigated, NOT implemented)

### 4a. Can our libass use a color emoji font (CBDT vs COLR)?
**No — neither format renders in color on libass 0.17.4.** Proven empirically:
Segoe UI Emoji (COLR/CPAL) *was* selected and *still* rendered monochrome.
- **CBDT/CBLC** (Noto Color Emoji): glyphs are embedded PNG bitmaps with no
  vector outline → libass renders **nothing** (blank).
- **COLR/CPAL** (Segoe UI Emoji, Noto COLR emoji): libass ignores the layer/palette
  table and traces only the base outline → **monochrome** (what we saw).
- libass has no released color-emoji support in the 0.17.x line. Getting color
  emoji out of libass is not a config/font change — it would require a different
  libass, which this static FFmpeg build does not offer.

### 4b. Fallback: a *purpose-built* monochrome emoji font (e.g. Noto Emoji)
- Bundle **Noto Emoji** (the monochrome sibling of Noto Color Emoji — clean
  single-color outlines/fills designed for one color) as a flat `.ttf` in
  `services/assets/fonts/captions/` so libass resolves it via `fontsdir`
  deterministically, **removing the host dependency in §3**.
- Result would look like what we already saw (flat, karaoke-tinted shapes) but
  cleaner and, crucially, **identical on Windows and Linux**.
- Cost is low and stays fully inside our invariants: still `ass=` + `fontsdir`,
  still single-pass, no new compositing. Caveat: the `fontsdir` scan is
  **non-recursive and flat** (see `DIAGNOSIS_FONTS.md` / `services/fonts.py`),
  and adding a broad-coverage font as an *unconditional* fallback could shift
  glyph resolution for other codepoints — it should be added as an explicit
  extra family, and the emoji still won't be *color*.

### 4c. Overlay path: emoji as PNG overlays composited at word positions
- Render each emoji as a color PNG (from a color emoji set) and
  `overlay`-composite it at the word's on-screen position/time.
- This is the **only** path that yields true color emoji, but it **directly
  breaks our single-pass compositing invariant** (`BURNIN_NOTES.md`: captions +
  overlays are burned in one `ass=` pass). Relaxing it would cost:
  - Computing per-word pixel rects for emoji (we position the *line* via
    `\an5\pos`, but libass does the intra-line word layout — we do not currently
    know each word's x/y; we'd have to measure or reproduce libass line layout,
    which is exactly the drift-prone WYSIWYG problem Stage 6 avoided).
  - Per-emoji, per-frame `overlay` inputs with karaoke-timed enable windows, or
    a second pass over the burned video — extra encode(s), more complexity, and
    a new class of preview/export drift.
  - Scaling/baseline-aligning the PNG to the caption font size per style.

## 5. Frontend preview — does it diverge from the burn? **Yes.**

The editor caption preview (`ElementBodies.jsx :: CaptionBody`) renders each word
as a DOM `<span>` with the raw text (`displayWord(w.id)` / `w.text`) — **no emoji
stripping** — and `fontFamily: '"Noto Sans Telugu", sans-serif'`. In every
browser, an emoji codepoint falls back to the **platform color emoji font**
(Segoe UI Emoji / Apple Color Emoji / Noto Color Emoji) and renders in **full
color**, and color emoji ignore the CSS `color` property.

So preview and burn diverge on two axes:
1. **Color vs flat:** preview shows a full-color emoji; the burn shows a flat,
   single-color outline (or blank/tofu on Linux).
2. **Karaoke tint:** in the preview the emoji keeps its natural colors through
   the whole karaoke sweep; in the burn the emoji shape *changes color*
   (yellow → grey → white) as the highlight passes. So an editor user who types
   an emoji sees a clean color emoji and gets something quite different burned in.

---

## Recommendation

Ranked by effort × invariant impact:

1. **Lowest effort, honest result — bundle Noto Emoji (monochrome) in `fontsdir`
   (§4b) and update the preview font stack to match.** This makes the output
   *deterministic across hosts* (fixes the blank/tofu-on-Linux landmine in §3),
   keeps the single-pass `ass=` invariant intact, and — critically — lets us make
   the **preview render the same monochrome font** so WYSIWYG holds. It does
   **not** give color emoji, but it makes the current behavior correct,
   predictable, and preview-accurate. This is the recommended first move.

2. **If product truly needs color emoji — the PNG-overlay path (§4c)** is the
   only way, and it must be scoped as a real feature: it breaks the single-pass
   invariant, needs per-word positioning we don't currently compute, and adds
   preview/export drift risk. Do not undertake casually.

3. **Do not** pursue "make libass render the color font" (§4a) — it is not
   possible on this FFmpeg/libass build; the color font is already being selected
   and still comes out monochrome.

**Meanwhile (no code change):** the safest interim guidance is that emoji in
caption text are effectively unsupported for burn — they render inconsistently
(legible-mono on the current host, blank/tofu elsewhere) and never match the
color emoji the editor preview shows.
