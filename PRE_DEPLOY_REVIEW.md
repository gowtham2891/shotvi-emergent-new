# Shotvi — Pre-Deployment Full-Application Review

Read-only audit of `shotvi-emergent-new/` across six roles (Backend, Frontend, GenAI, DevOps, QA, CEO). No code was changed. All `file:line` references are from the actual codebase as of this review.

---

## 1. Deployment Go / No-Go

**Verdict: NO-GO as-is — but the gap is a focused 1–2 week sprint, not a rebuild. The application layer is launch-quality; the deployment layer does not exist yet, and a handful of correctness/integrity bugs must be fixed before real users and real money touch it.**

### BLOCKER findings

1. **Same-video concurrent pipelines destroy each other's outputs** — `api/worker.py:127-131` — cleanup unlinks every `OUTPUT_DIR` file prefixed with `video_id` before cutting; nothing locks a video_id and the cache check (`api/main.py:115`) only matches `status=="done"`. Two users submitting the same YouTube URL (or one resubmit mid-run) run two full pipelines over the same files — job B's cleanup deletes job A's clips mid-caption, both write the same `_vertical`/`_captioned` paths concurrently. Corrupted or vanished deliverables with both jobs reporting "done".
2. **POST /jobs accepts local filesystem paths as pipeline input** — `api/main.py:147-152` + `api/worker.py:80-89` — `payload.url` is never validated as a URL; when `_extract_video_id` returns None, the raw string reaches the worker where `elif Path(url).exists()` treats ANY readable server path as an "upload". An authenticated user can submit `url="storage/uploads/<id>.mp4"` and process another user's private upload into their own clips — bypasses the entire by-video ownership model.
3. **Stale `activeJobId` traps the Upload page after the first success** — `frontend/src/pages/Upload.jsx:67-70` + `frontend/src/store/useAppStore.js:457-458` — `resetSubmission()` only runs on the failure path; after a completed job, revisiting /upload re-enters the processing view and `onDone` auto-navigates back to /clips. The user cannot start a second project without a hard refresh. Core-loop breakage.
4. **No Dockerfile and no compose services for the API or Celery worker** — `docker-compose.yml:1-36` defines only redis + indicxlit; `README.md:100-108` assumes a hand-run host venv. The target stack (four containers on one Hetzner CX31) does not exist. The image must also carry ffmpeg/ffprobe, node (yt-dlp js runtimes), libraqm (Pillow Telugu shaping), and the bundled fonts.
5. **`requirements.txt` cannot run the pipeline** — `requirements.txt:1-38` is missing `ctc-forced-aligner` (imported at `services/transcriber.py:1161`), `Pillow` (`services/overlay_renderer.py:47`), and `requests` (`api/main.py:405`). A clean container build installs green and crashes at transcription/overlay time.
6. **Live third-party secrets in plaintext `.env` with no secrets plan** — `.env:1-10` holds real values for GEMINI/SARVAM/RESEND/DEEPGRAM/SUPABASE/RAZORPAY keys (names only here). These must be provisioned server-side (env_file with 600 perms or Docker secrets) and rotated before go-live — the file has circulated on a dev machine, and the Razorpay webhook secret is additionally low-entropy/guessable (see Backend/DevOps).

## 2. Findings by role

### 2.1 Backend (FastAPI · Celery · Redis · FFmpeg)

Blockers #1 and #2 above originate here.

- [HIGH] Worker crash orphans jobs forever — `api/worker.py:22-26` — Celery runs on defaults: no `acks_late`, no `task_time_limit`, no watchdog. An OOM/SIGKILL mid-task drops the task silently; the job sits at "transcribing 20%" until its 24h TTL deletes it. No failure state, no retry.
- [HIGH] Sarvam batch transcription can hang a worker slot indefinitely — `services/transcriber.py:1438` — `job.wait_until_complete()` has no timeout; with `worker_prefetch_multiplier=1` and no Celery time limits, one stalled Sarvam job blocks the queue forever. Every ffmpeg call has a timeout; the most failure-prone external dependency has none.
- [HIGH] Billing webhook has no replay/ordering/duplicate protection — `api/main.py:886-934` + `api/billing.py:120-135` — signature verification is correct, but there is no `x-razorpay-event-id` dedup and no `created_at` comparison. Razorpay retries for up to 24h and delivery is unordered: a delayed `subscription.charged`/`.activated` arriving after `.cancelled` flips a cancelled user back to paid.
- [HIGH] Webhook-acked billing state can be lost by Redis — `docker-compose.yml:9-14` + `api/main.py:930` — RDB-snapshot-only persistence (no AOF); webhook returns 200 immediately after `hset`. A Redis crash inside the snapshot window permanently loses the plan flip and Razorpay never redelivers an acked event. No reconciliation job exists.
- [HIGH] Redis published on a host port with no auth — `docker-compose.yml:5-6` — `"6379:6379"` binds 0.0.0.0, no requirepass. On a non-firewalled host (and Docker bypasses ufw on Hetzner), jobs, owner stamps, and billing hashes are world-writable — including setting `plan=studio` directly, which `/billing/status` faithfully reports as server truth.
- [HIGH] Concurrent re-exports of the same clip collide on `_prepared.mp4` — `api/worker.py:302` — every other rerender artifact carries `job_suffix` (line 279) but the crop/trim intermediate does not; two exports of the same clip run `ffmpeg -y` against the same file concurrently and both burns read a torn intermediate.
- [HIGH] Crop stage processes every video in the shared outputs dir — `api/worker.py:136` → `services/vertical_cropper.py:201-214` — `crop_all_clips` supports a video_id filter the worker never passes; each job re-crops ALL raw clips in `storage/outputs`, including other users' (concurrent-write corruption risk), and cost grows with platform history instead of job size.
- [HIGH] Upload endpoint buffers whole video in RAM — `api/main.py:168-170` — `await file.read()` loads multi-GB files into memory, no size/content-type cap; a handful of concurrent uploads OOMs the API process. (Frontend sets `timeout: 0` for uploads, `frontend/src/api/uploads.js:40`.)
- [HIGH] Completion emails link to localhost from a sandbox sender — `api/email.py:9,13` — `http://localhost:5173/...` hardcoded (a port the frontend doesn't even use) and `onboarding@resend.dev` sandbox sender only delivers to the account owner. In prod, no real user gets email; failures are a swallowed print (`api/email.py:65-66`).
- [MEDIUM] Jobs marked done with silently broken clips — `api/worker.py:149-176` — cut/crop/caption failures print-and-continue (`services/video_cutter.py:394-407`); `output_clips` is built from whatever globs match, so a job where every cut failed still ends `done/100%` with `captioned_path: ""`.
- [MEDIUM] `/jobs/recover/{video_id}` grants ownership without any prior-access check — `api/main.py:179-246` — unlike every other by-video route it never calls `_require_video_access`; any authenticated caller who learns a video_id mints a done job over it and gains `/transcript` + `/clips/download` access. Real cross-user disclosure for private uploads.
- [MEDIUM] Job records expire in 24h and are never refreshed — `api/database.py:40` + `api/main.py:337` — `update_job` never extends the TTL; job lists empty daily, rerender records referencing exports vanish, and editor drafts (7-day TTL keyed by job_id) become unreachable because recovery mints a new job_id.
- [MEDIUM] `update_job` resurrects expired jobs as ownerless, TTL-less orphans — `api/database.py:56-60` — `hset` on an expired key recreates a partial hash with no owner/job_id/expiry: invisible to its user, permanent in Redis, and crashes `_job_to_out` (`api/main.py:794`) if listed.
- [MEDIUM] Non-atomic checkpoint writes can permanently poison a video — `services/transcriber.py:1925-1926`, `services/clip_selector.py:1830-1831` — transcript/clips JSON written with plain `open("w")` (unlike the correct tmp+`os.replace` at `api/main.py:377-380`); a crash mid-write leaves a truncated file every checkpoint guard treats as complete, failing all future runs for that video_id at `json.load`.
- [MEDIUM] `/thumbnails/{filename}` has no path containment — `api/main.py:82-87` — `OUTPUT_DIR / filename` served with no `resolve()`/`relative_to` check; on Windows, encoded `..\` segments escape OUTPUT_DIR and serve arbitrary files (e.g. `.env`) as image/jpeg. Safe on Linux only by accident. (The unauthenticated-media part is the documented known gap; the traversal is not.) The same containment used in `/clips/download` (`api/main.py:590-603`) — which was verified to hold — should be applied here.
- [MEDIUM] `video_id` from URL query is never sanitized — `api/main.py:743-767` — `?v=../../foo` survives `parse_qs` and flows into UPLOAD_DIR f-string joins and glob patterns (`main.py:187,368,690`; `worker.py:72`). Currently blocked by ownership checks/glob semantics, but no character allowlist exists for an identifier used in dozens of path constructions.
- [MEDIUM] Subscription create is check-then-act — `api/main.py:846-860` — two concurrent clicks both pass the "already active" guard and create two live Razorpay subscriptions; only the second's id is stored, the first keeps charging invisibly.
- [MEDIUM] Trim uses keyframe-snapped stream copy — `api/worker.py:453-466` — trim-only path puts `-ss` before `-i` with `-c:v copy`, snapping up to a full GOP (~2s); exported trim visibly disagrees with the frame chosen in the editor. The crop path re-encodes and is accurate.
- [MEDIUM] Overlay element props are unbounded user input — `services/overlay_renderer.py:136-148,295` + `api/models.py:112` — `elements` is `List[dict]` with no schema; scale/fontSize/width/height flow into PIL canvas dims and frame counts — a crafted element allocates gigapixel images and OOM-kills the shared render queue (authenticated DoS).
- [MEDIUM] Stale-cache regeneration deletes prior exports — `api/worker.py:127-131` — cleanup wipes ALL `{video_id}*` files including previous rerender outputs; those rerender jobs (and saved download URLs) 404 while still reading "done".
- [MEDIUM] API boots only if `storage/outputs` pre-exists — `api/main.py:51,61` — StaticFiles mount raises at startup on a fresh checkout/empty volume (only UPLOAD_DIR is created, `main.py:166`). Deploy-day crash.
- [MEDIUM] Redis full-keyspace scans on hot paths — `api/database.py:94-104,113-123,180-186` — `/jobs`, the POST cache check, every `/clips/download` and `/transcript` call iterate `scan_iter("job:*")` + `hgetall` over every job of every user; O(total jobs) per request.
- [LOW] `rerender_clip` crashes uglily when the source job expired — `api/worker.py:245-247` — `get_job` can return None between API validation and task execution; surfaces as `'NoneType' object has no attribute 'get'` to the user.
- [LOW] CORS wildcard with credentials — `api/main.py:43-49` — `allow_origins=["*"]` + `allow_credentials=True`; must become an explicit origin list before launch (not env-configurable today — a code change; see DevOps).
- [LOW] Progress-bar rotation silently dropped in burns — `services/overlay_renderer.py:161-167` — comment claims rotation is applied at compositing; `_composite_layers` (401-436) only overlays. Rotated bars burn unrotated.
- [LOW] yt-dlp fallback format breaks the `.mp4` assumption — `services/video_downloader.py:42,61` — the `best[height<=720]` fallback can yield `.webm` but `video_path` is hardcoded `{id}.mp4`; download "succeeds", then audio extraction fails on a missing file.
- [LOW] `except (ImportError, Exception): pass` swallows all romanization errors — `services/transcriber.py:1549-1554` — survivable (manual map follows) but hides real aksharamukha failures with zero logging.
- [LOW] `requirements.txt` mostly unpinned — yt-dlp/fastapi/celery/redis/mediapipe float free; a deploy-day install can pull breaking majors.
- [LOW] `pipeline.py` CLI wipes the entire shared outputs dir — `pipeline.py:313-318` — running the dev CLI on the prod box destroys all users' clips; scope it to a video_id even as a dev tool.
- [LOW] yt-dlp fetches remote challenge-solver JS at runtime — `services/video_downloader.py:47-48` — `'remote_components': {'ejs': 'github'}` pulls executable code from GitHub in the worker; pin for prod.
- [FEATURE] Stuck-job watchdog: periodic task failing jobs with a stale heartbeat (requires `update_job` to stamp a timestamp); `acks_late=True` + `task_time_limit` are two-line wins.
- [FEATURE] Webhook event ledger: `SETNX billing:event:{event-id}` + entity `created_at` comparison ≈ 10 lines for replay/ordering safety, and doubles as the audit trail the Postgres migration will want.
- [FEATURE] Nightly Razorpay reconciliation over `billing:sub:*` — self-heals both lost-webhook and out-of-order cases while Redis is the only billing store.

**Auth-guard route audit (explicitly requested):** every job- and billing-touching route carries `Depends(get_current_user)` with the 404-not-403 ownership rule via `_get_owned_job`/`_require_video_access` — /jobs (POST/GET/list/PATCH), rerender, draft save/load, /transcript/{video_id}, /transliterate, /tanglish, realign-line, metadata, /clips/download, /billing/status/subscription/cancel. The `/clips/download` path-traversal fix holds. Exceptions: `/` (health), `/billing/webhook` (HMAC by design), `/outputs` mount + `/thumbnails` (documented known auth gap — but /thumbnails additionally lacks path containment, flagged above), and two genuine gaps: `/jobs/recover/{video_id}` (authenticated but not ownership-scoped) and POST /jobs' local-path acceptance (Blocker #2).

**Assessment:** the auth/ownership layer is genuinely well-built — consistent 404 semantics, verified-JWT-only identity, owner stamped server-side — but it's undermined at the edges by two ingestion bypasses, and the flat shared outputs directory with no per-video locking makes concurrent same-video work mutually destructive. Job lifecycle has no crash resilience, and billing lacks the replay/durability protections a Redis-RDB-only, ack-once webhook store needs before real money flows. Clean key shapes and the existing reverse indexes mean the planned Postgres migration is structurally straightforward (single-module seam in `api/database.py`; jobs need `(owner, created_at)` and `(video_id, owner)` indexes).

**Assessment:** the core caption-editing machinery — the transcriptEdits delta model, lineSplits/realignment lockstep mirrors, undo/redo with gesture coalescing, graceful degradation of every AI seam — is genuinely well-engineered and backed by pure-logic tests. Fix the blocker plus the five HIGHs and clean out the fake affordances, and this is a shippable, unusually polished editor.

### 2.3 GenAI (Gemini · Sarvam ASR · MMS CTC · IndicXlit)

- [HIGH] Unvalidated Gemini score fields crash the whole job — `services/clip_selector.py:1345-1348,1363` — `float(gc.get("hook_score", 5.0))` / `int(gc.get("confidence_rank", 99))` with no guard; a non-numeric value ("8/10", "high") raises and fails the job after upload, transcription, and two paid Gemini calls already succeeded.
- [HIGH] Context cache TTL 300s vs multi-minute pipeline, no expiry fallback — `services/clip_selector.py:588,1246-1254,1507-1517` — cache created with `ttl="300s"`, but Call 1 runs thinking_level=HIGH on a full video (routinely minutes); on mid-pipeline expiry, Calls 2/3 still pass `cached_content` unconditionally (the inline fallback only triggers when creation failed at the start) — Call 2 raises "Fine cut API call failed" and kills the job; trim calls silently no-op.
- [HIGH] Prompt-injection surface: transcript inlined as undelimited instructions-adjacent text — `services/clip_selector.py:566-570,1152-1156,1258-1261` — third-party YouTube speech enters the prompt with no data/instruction fencing; second-order: Call-1 output (`why`/`visual_energy`) is re-embedded verbatim into the Call-2 prompt (`build_fine_cut_prompt:917-931`), giving injected content a second hop; LLM `why`/`hook_text` then reach output filenames (`services/video_cutter.py:360`) and user-visible metadata. Post-validation clamps sentence IDs but nothing constrains emitted text.
- [HIGH] Sarvam↔CTC fuzzy alignment tail collapse — `services/transcriber.py:1644-1676,1696-1697` — in the count-mismatch path, `ctc_idx` only advances on a match with LOOK_AHEAD=5; >5 consecutive unmatched CTC tokens (music, noise) leave every subsequent Sarvam word unmatched, and `_interpolate_missing`'s terminal-gap rule squeezes the entire remaining transcript into 0.5 seconds — silent garbage timestamps feeding clip selection and captions.
- [MEDIUM] Interpolation over negative gaps yields non-monotonic word times — `services/transcriber.py:1699-1706` — out-of-order fuzzy matches make `gap_duration` negative; interpolated words get decreasing/negative-duration timestamps with no clamp before karaoke grouping.
- [MEDIUM] JSON salvage can never recover fine-cut output — `services/clip_selector.py:254-273` — `_salvage_partial_json` keeps only depth-0 objects with a top-level `start_sent_id`, but fine-cut clips nest ids inside `segments`; a truncated Call-2 response salvages nothing and the job fails even when complete clip objects exist in the raw text.
- [MEDIUM] No retry/backoff on any production Gemini call — `services/clip_selector.py:1139-1178,1244-1283` — a single transient 429/503 fails the job; the only mitigation is disk checkpointing plus a manual re-run.
- [MEDIUM] File-URI checkpoint outlives Gemini File API expiry — `services/clip_selector.py:1758-1768` — `_file_uri.txt` reused forever with no `files.get()` liveness check; Gemini files expire (~48h), after which every regeneration run fails until the stale file is manually deleted.
- [MEDIUM] Empty/silent-audio transcripts spend Gemini money before failing — `services/transcriber.py:1967-1970` + `services/clip_selector.py:168-175,1735` — an empty Sarvam transcript still yields one `{start:0,end:0,text:""}` sentence that passes `load_transcript`'s only check; the pipeline then uploads the video and runs paid vision calls guaranteed to end in "Rough cut returned no candidates".
- [MEDIUM] Residual U+FEFF/U+200B bug class — zero hits for feff/200b/zero-width stripping in py/js. `services/caption_renderer.py:301` keeps a word whose text is U+FEFF (Python `strip()` doesn't remove it) while `frontend/src/api/transcripts.js:62-63` drops it (JS `trim()` does) → frontend and backend word index spaces diverge, mis-addressing every lineSplit/lineRealignment/wordEdit after that word. Also `services/transcriber.py:1530` (sentence-boundary check misses punctuation followed by an invisible char) and `:1539` (`_romanize_word` guarantees a fuzzy-match miss on ZW chars). U+200B-only words survive both sides and render as blank karaoke slots.
- [MEDIUM] `/realign-line` span unbounded → arbitrary-cost CTC runs — `api/main.py:474-475` + `services/realign_line.py:55-89` — only `line_end > line_start` is validated; a 10⁶s span extracts the entire audio and runs the MMS aligner per request on an authed endpoint, with the plausibility check rejecting only afterwards.
- [MEDIUM] Tanglish commit resolver routing/case gaps — `frontend/src/lib/lineEdit.js:15,87-91` + `EditableTranscript.jsx:305-310` — `LATIN_TOKEN=/^[A-Za-z]+$/` sends hyphenated/digit codemix ("mind-lo", "2nd", "100%") down the Telugu-as-is path (Latin silently stored as Telugu source); the known-tanglish reuse map is exact-case, so a capitalized unchanged word ("Deenni") round-trips through IndicXlit and can replace the original Telugu with a different spelling — the exact spurious-edit case the module's own contract forbids.
- [MEDIUM] Token tracker under-reports real cost — `exp/token_tracker.py:129-136` — `billable_output` excludes thinking tokens even though the summary prints "(billed as output)" (`:249`); with thinking_level=HIGH reported cost can be a small fraction of actual spend; cache-write cost never recorded.
- [MEDIUM] v20 tool architecture resends the full transcript on every flash-lite call — `exp/clip_tools.py:64-69` (sites at 155, 241, 324, 411, 493) — no context caching for TOOL_MODEL; the scout alone fires ~8 tool calls per window (`exp/clip_selector_v20.py:576-621`) so input tokens scale O(tool_calls × transcript).

## 4. Human visual gates still required

Automation cannot cover these; each is a concrete click-through only the owner can sign off.

**1. Tanglish quality read (the moat):**

## 5. Honest summary

The hard part of Shotvi is genuinely done and it shows: the Telugu caption pipeline, the WYSIWYG editor with its delta model and undo/redo, the Tanglish toggle, auth/ownership, and billing signature verification are careful, well-tested work — and the Telugu moat is on the surface of the product, not buried. But it is not deployable today for three separate reasons that compound: (1) the deployment layer does not exist — no Dockerfile or compose services for the API/worker, a requirements.txt that can't run the pipeline on Linux, secrets in a plaintext dev .env, and yt-dlp-from-a-datacenter-IP still an untested go/no-go unknown; (2) a small set of real correctness bugs will hit early users — the same-video pipeline mutual destruction, the local-path upload bypass, the Upload-page trap after the first project, the autosave/draft-wipe race, and the webhook replay/downgrade races around real money; and (3) the app is wrapped in a shell of fake UI (fictional three-tier pricing, invented social proof, dead share/billing/music/search affordances, no Terms/Privacy) that will make the first paying user distrust the honest core underneath. The shortest path to yes: fix the six blockers, run the DevOps gap list items 1–7 (with the yt-dlp test on the actual CX31 as the first gate), do the deletion-heavy honesty pass on the frontend shell, patch the five frontend HIGHs and the webhook dedup, then execute the five human visual gates above — realistically one focused sprint, after which this is a launchable, differentiated product rather than a liability.
